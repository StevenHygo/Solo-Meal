# 「一人食」微信小程序 v0 产品与技术设计

> 文档版本：v0.1  
> 日期：2026-07-21  
> 依据：[竞品与实现方式调研](./competitive-research.md)  
> 状态：第一版可实施设计，用于原型、研发、测试、上线与验收。  
> 核心约束：**v0 不包含社交功能，不依赖 LLM；核心找店链路在任何 AI 服务缺失时都能完整运行。**

## 0. 设计摘要

### 0.1 v0 要交付什么

v0 只解决一个闭环：

> 用户提供当前位置或搜索位置，按预算、距离、品类和单人友好条件筛选，获得一组有明确依据的餐厅，查看详情后导航到店。

为保证第一版能够稳定上线，v0 只依赖三类数据：

1. 地图服务提供的 POI、地址、品类、营业信息和距离；
2. 运营人员核验的单人座位、单人份、最低消费、用餐时长等结构化数据；
3. 用户提交的收藏和“信息不准确”反馈。反馈不形成公开动态、关注关系或社交评价流。

### 0.2 v0 明确不做什么

- 不抓取或聚合小红书、抖音、大众点评等平台内容；
- 不提供关注、好友、评论区、公开清单、拼饭或陌生人社交；
- 不使用 LLM 解析查询、生成推荐、总结评价或处理截图；
- 不做公开长文本评价和图片 UGC；
- 不自建支付、团购、排队、预订、外卖或 POS；
- 不承诺实时座位和实时排队；
- 不为追求全国覆盖而牺牲单个商圈的数据可信度。

### 0.3 技术总原则

- **简单但不耦合：** v0 使用模块化单体，不提前拆微服务，但模块之间只通过明确接口和领域事件协作。
- **确定性优先：** 搜索和排序使用可测试、可解释的规则；未知数据不由模型猜测。
- **供应商可替换：** 地图、消息、存储和未来 AI 能力放在适配层，业务代码不直接依赖供应商响应结构。
- **API 优先：** 小程序只是第一个客户端；业务能力通过版本化 API 提供，为未来 iOS/Android App 复用。
- **安全默认开启：** 最小权限、输入校验、密钥隔离、审计日志、限流和隐私最小化不是上线后的补丁。
- **故障可降级：** 地图供应商、缓存、非核心写入或分析系统故障时，核心搜索尽量继续提供受限服务。

## 1. 产品目标

### 1.1 一句话定位

> 帮用户快速找到“一个人也方便、吃得合适、现在去得了”的附近餐厅，并说明推荐依据。

### 1.2 用户问题

普通地图能回答“附近有什么”，但很难直接回答：

1. 一个人去是否容易入座？
2. 是否有吧台、单人桌或适合单人的普通桌？
3. 是否有单人份、最低消费是否合适？
4. 能否在预算和时间内完成这顿饭？
5. 页面信息最近什么时候核验，可信程度如何？

### 1.3 设计原则

- 首屏直接找店，不做内容资讯首页；
- 预算是筛选项，不把“一人食”等同于廉价快餐；
- 显示证据和更新时间，不只显示一个总分；
- 拒绝定位后仍能搜索商圈、地铁站或地址；
- 不注册也可搜索和浏览，登录只在跨设备收藏等必要场景触发；
- 结果不足时建议用户放宽条件，不静默取消硬筛选；
- 所有核心字段支持 `unknown`，不把“未知”显示为“否”。

## 2. 目标用户与核心场景

### 2.1 v0 核心用户

| 用户 | 任务 | 主要决策条件 |
| --- | --- | --- |
| 上班族 | 30-45 分钟内完成午餐或晚餐 | 距离、预算、出餐速度、排队风险 |
| 独自生活者 | 找可以反复去的日常餐厅 | 单人座位、点餐压力、价格、安静程度 |
| 临时落单者 | 立即在附近决定去哪吃 | 无需注册、少输入、导航准确 |
| 独自旅行者 | 在陌生区域找接受 1 人的餐厅 | 单人接待、位置、品类、营业状态 |

有复杂饮食限制的用户是后续重点人群。v0 可提供素食等基础标签，但只有在数据可核验时才展示，不提供医疗或过敏安全保证。

### 2.2 核心用户故事

- 我想找步行 10 分钟、50 元以内、一个人可以快速吃完的面馆。
- 我不想拼桌，希望优先看到有吧台或单人桌的店。
- 我拒绝了定位权限，仍想通过“静安寺”搜索附近餐厅。
- 我想知道“单人友好 85 分”具体来自哪些已核验信息。
- 我发现餐厅已经不接待单人，希望能快速报错。
- 我想收藏餐厅，并在换手机后通过微信登录恢复收藏。

## 3. 成功指标与服务目标

### 3.1 北极星指标

**每周有效找店决策数（WSD）**：一次搜索后，用户查看餐厅详情并点击导航、电话或复制地址中的任一动作，按用户、餐厅和自然日去重。

v0 不使用后台持续定位推测用户是否真实到店。

### 3.2 产品指标

| 指标 | 定义 | 试点目标 |
| --- | --- | ---: |
| 3 分钟决策率 | 搜索后 3 分钟内触发导航/电话/复制地址 | >= 40% |
| 有效结果覆盖率 | 查询至少返回 3 家中/高置信候选 | >= 80% |
| 详情转行动率 | 详情页到导航/电话/复制地址 | >= 30% |
| 信息纠错率 | 每 100 次详情浏览收到的有效报错 | 持续下降 |
| 次周复用率 | 首周完成找店决策的用户次周再次使用 | >= 20% |
| 定位拒绝可用率 | 拒绝定位后仍成功获得结果 | >= 95% |

这些是试点假设，不是行业基准。上线两周后按真实基线调整。

### 3.3 v0 稳定性目标

| 指标 | 目标 | 说明 |
| --- | --- | --- |
| 核心 API 月可用性 | >= 99.5% | 排除约定维护窗口 |
| 搜索 API P95 | <= 1.5 秒 | 命中缓存或主库，不含首次地图供应商慢请求 |
| 搜索端到端 P95 | <= 3 秒 | 包含地图供应商调用 |
| 详情 API P95 | <= 800 毫秒 | 正常负载 |
| 服务端 5xx 比例 | < 1% | 按 5 分钟窗口告警 |
| 发布回滚时间 | <= 15 分钟 | 应用版本，不含不可逆数据修复 |
| 数据恢复目标 | RPO <= 15 分钟，RTO <= 2 小时 | 使用支持 PITR 的托管数据库 |

## 4. v0 信息架构

### 4.1 底部导航

保持 3 项，不包含社交入口：

1. **发现：** 位置、搜索、筛选、列表/地图；
2. **收藏：** 用户私有收藏，未登录时保存在本机；
3. **我的：** 登录、默认预算、常用位置、隐私、反馈和关于。

### 4.2 页面清单

| 页面 | 主要任务 | 是否必须登录 |
| --- | --- | --- |
| 发现首页 | 选择位置、场景和筛选 | 否 |
| 搜索位置 | 搜索商圈、地铁站、地址或地图选点 | 否 |
| 结果列表/地图 | 比较餐厅、切换地图 | 否 |
| 餐厅详情 | 查看证据、营业信息、导航 | 否 |
| 收藏 | 本机或账号收藏 | 跨设备同步时需要 |
| 信息纠错 | 报告关闭、价格或单人接待变化 | 建议登录，允许匿名限频提交 |
| 我的 | 偏好、隐私、清除数据、退出登录 | 部分功能需要 |
| 运营后台 | POI 合并、资料核验、审核纠错 | 运营 RBAC |

### 4.3 首次使用流程

```text
打开小程序
  -> 展示搜索入口和“使用当前位置”按钮
  -> 用户主动点击后解释定位用途
  -> 同意：获取一次定位
     拒绝：进入手动位置搜索
  -> 选择场景/筛选
  -> 查看列表或地图
  -> 查看详情
  -> 导航/电话/复制地址
```

不得在用户尚未表达附近搜索意图时立即申请精确定位。

### 4.4 找店流程状态

| 状态 | UI 行为 |
| --- | --- |
| 未选择位置 | 显示位置输入，不请求餐厅数据 |
| 定位中 | 保持页面结构稳定，显示局部加载状态 |
| 定位拒绝 | 显示手动搜索入口，不展示错误恐吓文案 |
| 搜索中 | 保留筛选与上次结果，避免全屏空白 |
| 有结果 | 默认列表，允许地图切换 |
| 条件过严 | 明确展示哪个条件导致结果少，建议逐项放宽 |
| 地图供应商超时 | 展示缓存结果并标记“部分信息可能不是最新” |
| 完全无数据 | 提供更换位置、扩大范围和反馈入口 |

## 5. v0 功能需求

### 5.1 P0：上线必须完成

#### A. 位置与地理搜索

- 一次性当前位置；
- 商圈、地铁站、地址关键词搜索；
- 地图选点和“搜索此区域”；
- 常用位置由用户主动命名，不在界面展示完整家庭/公司地址；
- 统一使用 GCJ-02 坐标；
- 搜索半径支持 500 米、1 公里、3 公里和 5 公里。

#### B. 筛选

- 距离或步行时间；
- 预算：`<=30`、`30-60`、`60-100`、`100+`、自定义上限；
- 品类/菜系；
- 营业中；
- 单人座位/吧台；
- 有单人份或无明确最低消费；
- 快速用餐；
- 安静程度；
- 基础饮食标签，仅展示已核验值。

预算、营业中和用户明确选择的饮食条件可设为硬条件；安静、快速等默认作为排序偏好。

#### C. 结果列表与地图

每个结果固定展示：

- 名称、主品类、距离和营业状态；
- 预算区间；
- 单人友好分和可信度等级；
- 最多 3 条依据，例如“有吧台”“支持单人份”“运营于 7 天前核验”；
- 数据最后核验时间；
- 收藏和导航入口。

结果卡片不得展示没有来源的“热门”“必吃”“最适合”等绝对文案。

#### D. 餐厅详情

按决策顺序展示：

1. 当前营业状态、距离和行动按钮；
2. 单人友好分、可信度和匹配当前筛选的原因；
3. 接待 1 人、座位类型、单人份、最低消费、预计用餐时长、安静程度；
4. 人均区间、主品类和已核验标签；
5. 地址、电话、营业时间和地图导航；
6. 来源、核验时间和“信息不准确”入口。

“预计用餐时长”“安静程度”等非地图标准字段必须展示为区间或等级，并注明核验来源，不伪装成实时值。

#### E. 私有收藏

- 未登录用户收藏到本机；
- 微信登录后可同步到服务端；
- 首次登录时将本机收藏幂等合并到账户；
- 支持取消收藏和按距离排序；
- v0 收藏没有公开、关注、点赞、转发或协作属性。

#### F. 信息纠错

纠错类型固定为：

- 店已关闭/搬迁；
- 不接待单人；
- 营业时间错误；
- 价格区间错误；
- 单人座位或最低消费信息错误；
- 分店匹配错误；
- 其他。

单条用户报告不会直接覆盖餐厅事实；它只创建运营复核任务。相同用户、设备和餐厅的重复报告需要幂等合并和限频。

纠错备注为可选纯文本，最多 200 个字符；不支持 HTML、脚本、外链、图片和文件上传，服务端仍执行内容长度、字符集和敏感信息校验。

#### G. 运营后台

- 运营账号的角色和权限管理；
- 地图 POI 导入与供应商字段对比；
- 连锁分店去重、合并和拆分；
- 单人字段录入、证据附件和核验日期；
- 用户纠错任务处理；
- 每次修改的前后值、操作者、时间和原因审计；
- 数据发布支持草稿、审核、发布和撤回状态。

### 5.2 P1：v0 稳定后再做

- 结构化到店反馈及达到样本阈值后的匿名聚合；
- 商家认领和商家自助维护；
- 第三方预订/排队合作；
- 更精细的个性化排序；
- 多城市、多语言和无障碍增强；
- 可授权的社交内容导入；
- 可插拔的 LLM 搜索理解和证据摘要；
- iOS/Android App 客户端。

P1 功能只有在独立开关启用后才生效，不能改变 v0 API 的既有语义。

## 6. 单人友好数据标准

### 6.1 核心字段

| 维度 | 字段 | 类型 | v0 来源 |
| --- | --- | --- | --- |
| 接待 | `accepts_solo` | `yes/no/unknown` | 运营核验 |
| 高峰政策 | `solo_peak_policy` | 枚举 + 文本备注 | 运营核验 |
| 座位 | `seat_types` | 多选枚举 | 运营核验 |
| 吧台数量 | `counter_seats` | 非负整数/unknown | 运营核验 |
| 单人份 | `solo_portion_available` | `yes/no/unknown` | 运营核验 |
| 最低消费 | `minimum_spend` | 金额/none/unknown | 运营核验 |
| 价格 | `solo_price_min/max` | 金额区间 | 地图 + 运营核验 |
| 用餐时长 | `meal_duration_min/max` | 分钟区间/unknown | 运营核验 |
| 氛围 | `noise_level` | 1-5/unknown | 运营核验 |
| 营业时间 | `opening_hours` | 周期时间段 | 地图 + 运营复核 |
| 品类 | `cuisine_codes` | 标准枚举 | 地图 + 运营复核 |
| 核验 | `last_verified_at` | 时间戳 | 系统生成 |

所有金额以最小货币单位整数保存，所有时间以带时区时间戳保存。业务层禁止用浮点数存金额。

### 6.2 证据模型

餐厅字段和证据分离，便于未来加入商家、用户、第三方或 AI 来源：

```json
{
  "evidence_id": "ev_01J...",
  "restaurant_id": "res_01J...",
  "attribute": "counter_seats",
  "value": 8,
  "source_type": "operator_verified",
  "source_ref": "task_01J...",
  "observed_at": "2026-07-18T12:30:00+08:00",
  "expires_at": "2027-01-18T12:30:00+08:00",
  "confidence": 0.90,
  "status": "published",
  "schema_version": 1
}
```

v0 允许的 `source_type`：`map_provider`、`operator_verified`、`user_report`。未来新增来源只能扩充枚举和适配器，不改变餐厅主表语义。

### 6.3 新鲜度规则

| 字段 | 建议复核周期 |
| --- | ---: |
| 营业/关闭状态 | 30-90 天 |
| 是否接待单人、高峰政策 | 90 天 |
| 价格、最低消费、单人份 | 120 天 |
| 座位结构 | 180 天 |
| 品类和地址 | 180 天或供应商变更时 |

过期证据不立即删除：排序时降低可信度，界面显示“信息较久未核验”，并进入运营待复核队列。

## 7. v0 搜索与排序

### 7.1 处理流程

```text
规范化位置和筛选
  -> 查询本地餐厅库（PostGIS）
  -> 缺少覆盖时调用地图 Provider Adapter 补充 POI
  -> 去重并持久化供应商映射
  -> 应用硬条件
  -> 计算规则分和可信度
  -> 稳定排序
  -> 返回结果与可解释原因
```

地图供应商只负责召回和基础 POI，不负责最终单人友好排序。

### 7.2 单人友好分

v0 使用配置化规则，权重有版本号并可回滚：

| 分项 | 权重 |
| --- | ---: |
| 明确接待 1 人 | 25 |
| 有单人桌/吧台/适合单人的座位 | 25 |
| 有单人份或无最低消费 | 20 |
| 价格与单人点餐便利 | 10 |
| 用餐时长 | 10 |
| 安静程度 | 5 |
| 信息新鲜度 | 5 |

仅对已知字段归一化，未知字段降低可信度而不是直接计 0。餐厅明确不接待单人时从候选中排除。

### 7.3 查询相关性

```text
rank_score = 0.35 * solo_fit
           + 0.25 * distance_fit
           + 0.15 * budget_fit
           + 0.15 * cuisine_fit
           + 0.10 * time_fit
```

- 权重存储在版本化配置中，不硬编码在客户端；
- 同分时按可信度、距离、稳定餐厅 ID 排序，保证翻页稳定；
- 每次响应返回 `ranking_version`，便于回放问题；
- v0 不使用用户画像、协同过滤或机器学习模型。

### 7.4 可信度

可信度独立于友好分：

- `high`：核心字段由运营近期核验，且来源无冲突；
- `medium`：部分核心字段已核验，或证据接近过期；
- `low`：主要依据地图品类/店型，缺少单人字段；
- `unknown`：无法形成有效判断，不显示精确友好分。

前端只展示后端返回的原因代码和本地化文案，不自行重新计算分数。

## 8. 技术架构

### 8.1 技术选型

| 层 | v0 选择 | 选择理由 |
| --- | --- | --- |
| 小程序 | 微信原生小程序 + TypeScript | 当前单端，地图、授权和审核链路直接 |
| API | NestJS + TypeScript | 模块化、验证和 OpenAPI 工具成熟，前后端语言统一 |
| 数据库 | PostgreSQL + PostGIS | 空间查询、事务和未来复杂数据关系 |
| 缓存/限流 | Redis | 热点查询、分布式限流和短期会话 |
| 对象存储 | 腾讯云 COS 或同区域替代 | 运营证据附件和导出文件 |
| 异步任务 | 托管队列或数据库 Outbox Worker | POI 同步、复核任务和非核心事件 |
| 地图 | 腾讯位置服务 Provider Adapter | 微信生态和中国大陆坐标一致性 |
| 部署 | 中国大陆合规云、容器化部署 | 可重复发布、水平扩容、备案和网络时延 |
| 可观测性 | 指标 + 结构化日志 + Trace + 错误跟踪 | 稳定性和问题定位 |

Skyline 仅在兼容性验证通过后启用；业务组件不得依赖只能在单一渲染引擎运行的非必要能力。

### 8.2 为什么选择模块化单体

v0 团队和流量尚未证明需要微服务。过早拆分会增加网络故障、部署、鉴权和数据一致性成本。模块化单体仍要求边界清晰：

```text
clients/
  wechat-miniapp/
  admin-web/
packages/
  api-contracts/        # OpenAPI 生成类型、错误码、DTO
  domain-types/         # 无框架依赖的领域类型
server/
  modules/
    identity/
    location/
    restaurant/
    search/
    ranking/
    favorite/
    feedback/
    operations/
    audit/
  infrastructure/
    database/
    cache/
    map-providers/
    queue/
    storage/
    observability/
  app/
```

约束：

- 模块不得直接读取其他模块的数据库表；通过应用服务接口或领域事件访问；
- `domain-types` 不依赖 NestJS、微信 SDK 或地图 SDK；
- 供应商原始响应只能存在于 `infrastructure/map-providers`；
- 数据库迁移由服务端仓库统一管理；
- 每个模块有自己的单元测试、契约测试和所有者；
- 当某模块有独立扩容或隔离需求时，再按边界拆为服务。

### 8.3 运行结构

```text
微信小程序 / 未来 App / 运营后台
                 |
             HTTPS API
                 |
        API Gateway / WAF / 限流
                 |
          模块化单体服务
       |         |          |
 PostgreSQL   Redis      Outbox Worker
       |                    |
   只读副本(后续)      地图 Provider Adapter
                            |
                       腾讯位置服务
```

运营后台与用户 API 使用不同入口、认证策略和权限，不能只依赖前端隐藏按钮。

### 8.4 面向 App 的客户端边界

- 小程序不直接访问数据库、地图 WebService 或对象存储私有接口；
- 搜索、收藏、纠错和配置均走稳定 API；
- UI 展示模型由 API 返回，客户端只负责交互与缓存；
- 通用 API 不携带微信组件或页面路径概念；
- 身份模块通过 `identity_provider` 支持 `wechat_miniapp`，未来可增加 Apple、手机号等登录方式；
- 深链、推送和应用内购买属于客户端适配，不进入餐厅/排序领域模块。

### 9.2 餐厅去重

按顺序执行：

1. 同供应商 POI ID 精确匹配；
2. 标准化电话精确匹配；
3. 100 米范围内的标准化名称 + 地址相似匹配；
4. 连锁品牌保留具体分店，不按品牌名合并；
5. 低置信候选进入人工合并任务；
6. 合并操作保留别名和重定向，已收藏的旧 ID 不失效。

### 9.3 数据迁移原则

- 迁移脚本只向前执行并纳入代码评审；
- 发布采用 expand/contract：先加兼容字段，再迁移读写，最后删除旧字段；
- 大表回填异步分批执行，不在发布事务中长时间锁表；
- 每次迁移在预发布环境使用接近生产规模的数据演练；
- 破坏性迁移必须有备份、回滚或恢复方案。

## 10. API 设计

### 10.1 约定

- 基础路径：`/api/v1`；
- OpenAPI 是接口契约单一来源，并生成小程序/App 类型；
- 所有响应携带 `request_id`；
- 时间使用 ISO 8601 和明确时区；
- 金额使用整数最小货币单位；

- 错误使用稳定错误码，不把内部异常栈返回客户端；
- 不在既有版本中改变字段含义，破坏性变化发布新 API 版本。

## 11. 安全设计

### 11.1 信任边界

- 小程序、App 和运营后台都属于不可信客户端；
- 所有筛选、ID、价格、坐标、游标和状态都在服务端校验；
- 不能依赖客户端隐藏字段或按钮实现权限控制；
- 地图供应商响应属于外部输入，必须校验类型、范围和长度；
- 运营附件属于不可信文件，上传后经过类型、大小和恶意内容检查。

### 11.2 身份与会话

- `wx.login` 的临时 `code` 只发送到自有服务端；
- `code2Session` 等平台密钥只存在于服务端 Secret Manager；
- 外部身份与内部 `user_id` 分离，业务表不直接使用 `openid`；
- `openid/unionid` 加密或隔离存储，日志中禁止输出；
- 用户会话短期有效、可撤销，退出和删号立即失效；
- 运营后台强制 MFA，使用 RBAC 和最小权限；
- 高风险操作需要重新认证，并写入不可变审计日志。

### 11.3 API 与应用安全

- 全链路 HTTPS，启用 HSTS、合理 TLS 版本和安全响应头；
- API Gateway/WAF 限制异常流量和常见攻击；
- DTO 白名单校验，拒绝未知或超长字段；
- 数据访问全部使用参数化查询/ORM，不拼接 SQL；
- 输出编码，运营后台防 XSS、CSRF 和点击劫持；
- 匿名搜索按 IP/设备粗粒度限流，登录写操作按用户限流；
- 游标和分页令牌签名，客户端不得篡改排序边界；
- 写接口使用幂等键防重放；
- 管理 API 与用户 API 分离域名或路由，并限制网络来源；
- 依赖锁定版本，CI 执行 SCA、Secret Scan 和镜像扫描；
- 生产数据库和缓存不暴露公网。

### 11.4 密钥和配置

- 所有密钥进入云 Secret Manager，不进入代码、镜像或前端包；
- 开发、预发布、生产使用独立账号、数据库、地图 Key 和存储桶；
- 地图 Key 配置调用签名、域名/IP 白名单、最小配额和费用告警；
- 密钥定期轮换，泄漏时可独立吊销，不需要全量发版；
- 功能开关分环境管理，修改有审批和审计。

### 11.5 隐私与数据最小化

- 定位仅在用户主动点击后申请，一次使用，不申请后台持续定位；
- 精确坐标只用于当前搜索，业务日志默认记录网格或商圈，不记录完整轨迹；
- 常用位置默认只保存用户自定义名称和加密坐标；
- 不收集通讯录、好友关系、微信昵称或头像，除非未来功能有明确必要；
- 分析事件不包含手机号、外部身份、精确位置或自由文本；
- 明确保留周期：原始请求日志建议 7-14 天，聚合指标长期保存，按合规评审调整；
- 用户可以查看、导出或删除其收藏、偏好和账号；
- 微信定位与隐私接口按[小程序用户隐私保护指引](https://developers.weixin.qq.com/miniprogram/dev/framework/user-privacy/)配置。

## 12. 稳定性与故障处理

### 12.1 超时预算

一次搜索建议的服务端预算：

| 环节 | 超时 |
| --- | ---: |
| Redis 读取 | 100 毫秒 |
| 主数据库空间查询 | 500 毫秒 |
| 地图供应商 | 1,200 毫秒 |
| 整体搜索请求 | 2,500 毫秒 |

客户端在 3 秒后展示明确的受限状态，不无限等待。实际数值在压测后校准。

### 12.2 重试、熔断与降级

- 只对幂等读取和明确可重试错误自动重试；
- 单次请求最多重试 1 次，采用指数退避和随机抖动；
- 地图供应商连续失败后开启熔断，避免拖垮线程池；
- 熔断时返回本地库/缓存结果，并标记数据新鲜度；
- Redis 故障时回源数据库，不让缓存成为单点；
- 分析埋点、审计导出等非核心写入异步处理，失败不阻塞找店；
- 排序配置读取失败时使用最后一个已发布且校验通过的版本；
- 客户端通过远程开关关闭地图视图、收藏同步等非核心功能。

### 12.3 缓存策略

- 缓存键包含位置网格、筛选摘要、排序版本和数据版本；
- 热门商圈搜索短 TTL，并使用 `stale-while-revalidate`；
- 餐厅资料发布后通过事件主动失效相关缓存；
- 不缓存用户敏感数据到共享公共缓存；
- 防止缓存穿透、击穿和雪崩，TTL 加随机抖动；
- 任何缓存数据都可由主库重建。

### 12.4 数据一致性

- 收藏使用唯一约束和幂等写；
- 纠错创建任务和审计日志在同一数据库事务中完成；
- 业务事务和异步事件通过 Outbox 同事务提交；
- 消费者至少一次投递，处理器必须幂等；
- 不使用分布式事务连接地图供应商；外部同步失败进入可重试任务。

### 12.5 备份与恢复

- 托管 PostgreSQL 开启多可用区和时间点恢复；
- 每日全量备份 + 连续日志归档；
- 对象存储开启版本控制和生命周期；
- 每季度执行恢复演练，记录实际 RPO/RTO；
- 审计日志单独备份并限制删除权限；
- 恢复流程写入运行手册，不依赖单个工程师记忆。

## 13. 可观测性与运行手册

### 13.1 指标

必须监控：

- 请求量、成功率、P50/P95/P99 延迟；
- 搜索无结果率、低置信结果率、地图回源率；
- 地图供应商延迟、错误率、配额和费用；
- PostgreSQL 连接、慢查询、锁等待、复制延迟；
- Redis 命中率、内存和逐出；
- 队列积压、重试和死信；
- 5xx、客户端版本分布和崩溃率；
- 产品漏斗与定位拒绝后的成功率。

### 13.2 日志与 Trace

- 使用结构化 JSON 日志；
- 每个请求贯穿 `request_id` 和 Trace ID；
- 对精确位置、外部身份、令牌和密钥做脱敏/禁止记录；
- 错误日志记录错误码、模块和依赖状态，不记录用户敏感正文；
- 日志访问受 RBAC 控制并有保留周期；
- 搜索问题可通过 `query_id + ranking_version + data_version` 回放。

### 13.3 告警分级

| 级别 | 示例 | 响应 |
| --- | --- | --- |
| P0 | 核心搜索完全不可用、数据泄漏 | 立即通知，停止发布，启动应急预案 |
| P1 | 5xx 持续超阈值、数据库主节点故障 | 15 分钟内响应 |
| P2 | 地图回源失败但缓存可用、队列积压 | 工作时间处理并跟踪 |
| P3 | 单餐厅数据错误、非核心统计延迟 | 进入常规任务 |

每类告警必须关联负责人、看板和处理手册，避免只有告警没有动作。

## 14. 测试策略

### 14.1 自动化测试层级

| 测试 | 覆盖重点 | 门禁 |
| --- | --- | --- |
| 单元测试 | 排序、营业时间、金额、坐标、权限规则 | PR 必须通过 |
| 模块集成测试 | PostgreSQL/PostGIS、Redis、事务、Outbox | PR 必须通过 |
| Provider 契约测试 | 地图请求/响应映射、异常和限流 | 每日 + 发版前 |
| API 契约测试 | OpenAPI 向后兼容、错误码 | PR 必须通过 |
| 小程序 E2E | 定位允许/拒绝、搜索、筛选、详情、收藏 | 发版前 |
| 安全测试 | 越权、注入、XSS、限流、密钥扫描 | 发版前 |
| 性能测试 | 搜索并发、热点商圈、缓存失效 | 首发及重大变更前 |
| 恢复测试 | 数据库恢复、缓存重建、回滚 | 季度 |

排序规则必须有固定测试数据集，任何权重修改都比较 Top N 变化并需要产品确认。

### 14.2 必测异常场景

- 用户拒绝定位或系统定位关闭；
- 地图供应商超时、返回空数据、返回畸形字段或达到配额；
- Redis 不可用；
- 数据库慢查询或连接池耗尽；
- 重复点击收藏/纠错；
- 客户端断网、弱网、请求超时和重试；
- 连锁同名店和相邻门店错配；
- 跨午夜营业、节假日营业、时区和夏令时数据；
- 旧客户端遇到新增字段或功能关闭；
- 删除账号后再次登录。

### 14.3 v0 验收标准

- 定位同意和拒绝两条路径都能完成找店；
- 搜索位置、地图选点和当前位置的坐标语义一致；
- 硬筛选不会被静默取消；
- `unknown` 不显示成“否”或虚假精确值；
- 每个友好分都能展开到来源和核验时间；
- 导航到正确分店，旧餐厅 ID 合并后仍正确重定向；
- 收藏重复提交不产生重复数据；
- 用户纠错不会直接污染公开资料；
- 地图、Redis 和非核心队列分别故障时通过降级测试；
- 生产备份已执行一次真实恢复演练；
- 无阻断级安全问题，高危漏洞为 0；
- 生产依赖清单、部署清单和调用图中不存在社交平台 SDK、社交内容服务或 LLM 服务；
- 小程序主流 iOS/Android 微信版本真机测试通过；
- 监控、告警、回滚和应急联系人已配置。

## 15. 发布、灰度与回滚

### 15.1 环境

- `development`、`staging`、`production` 完全隔离；
- 预发布使用脱敏或合成数据，不复制生产精确位置/身份；
- 基础设施使用 IaC 或可审计配置管理；
- 生产变更必须经过代码评审和 CI 门禁。

### 15.2 发布流程

```text
PR 检查
  -> 单元/集成/契约/安全扫描
  -> 预发布部署和 E2E
  -> 数据库兼容性检查
  -> 后端小流量灰度
  -> 小程序体验版/灰度
  -> 观察核心指标
  -> 全量或回滚
```

### 15.3 回滚规则

- 应用镜像保留最近稳定版本，一键回滚；
- 数据迁移必须向后兼容，避免应用回滚后无法读取；
- 排序配置和功能开关可独立回滚，不需要重新发版；
- 发布后若 5xx、P95、无结果率或崩溃率越过阈值，自动暂停扩大流量；
- 回滚后创建事件复盘，不以“恢复了”代替根因修复。

## 16. 冷启动与运营

### 16.1 试点范围

选择一个城市的 1-3 个相邻高密度商圈。扩张门槛按商圈计算，而不是按城市名称：

- 核心时段可返回至少 3 家中/高置信候选；
- 至少 100 家餐厅完成单人字段核验；
- 高曝光餐厅 90 天内核验覆盖达到目标；
- 连锁分店错配率和有效纠错率达到可控水平。

### 16.2 首批数据流程

1. 从地图召回约 1,000 家餐饮 POI；
2. 根据品类、距离和营业状态筛出约 200 家候选；
3. 线下、电话或商家公开资料核验 100 家；
4. 双人复核高曝光餐厅和易混淆分店；
5. 发布后按曝光量、证据过期和用户纠错生成核验任务；
6. 每周抽样复核已发布数据，记录准确率。

v0 不使用现金或优惠换公开好评，因为 v0 没有公开评价功能。

## 17. 开发计划

### 阶段 0：产品与数据验证，2 周

- 访谈 15-20 位目标用户；
- 选定试点商圈；
- 冻结 v0 字段字典和评分规则；
- 人工核验 100 家餐厅；
- 用交互原型完成至少 30 次找店任务；
- 完成威胁建模和数据隐私评审。

**退出条件：** 至少 70% 测试用户认为单人字段比普通地图列表更有决策价值。

### 阶段 1：v0 开发，6-8 周

- 小程序位置、搜索、筛选、列表/地图和详情；
- 模块化后端、PostGIS、地图适配器和规则排序；
- 私有收藏、纠错和运营后台；
- 安全基线、监控、告警、备份和发布流水线；
- 自动化测试和故障降级演练。

### 阶段 2：灰度与稳定性，2 周

- 100-500 名试点用户灰度；
- 修正 POI 错配、排序和数据过期问题；
- 容量压测、恢复演练和安全复测；
- 达到 v0 验收门槛后全量发布试点区域。

## 18. 未来演进

### 18.1 社交和 LLM 如何接入

未来能力必须作为独立模块加入：

```text
content-import module
  -> 接收用户主动授权的链接/截图
  -> 提取候选餐厅
  -> 调用 restaurant module 完成 POI 匹配

ai-gateway module
  -> 自然语言转结构化筛选
  -> 基于 evidence ID 生成可引用摘要
  -> 超时/错误时回退到 v0 表单和结构化详情

social module
  -> 公开清单、关注和分享权限
  -> 只引用 restaurant_id，不复制餐厅主数据
```

这些模块不能绕过证据、权限、内容授权和审核规则，也不能让搜索服务依赖 LLM 才能工作。

### 18.2 升级为 App

升级为 iOS/Android App 时保留：

- 同一版本化 API 和领域模型；
- 同一餐厅 ID、证据、排序和收藏数据；
- 同一鉴权核心，通过身份适配器增加 Apple/手机号等方式；
- 同一远程配置、埋点字典和错误码；
- 同一安全、监控和发布后端。

需要新增的是客户端设计系统、原生地图/定位适配、深链、推送、应用商店发布和移动端安全加固，而不是重写后端业务。

### 18.3 何时拆微服务

只有满足下列条件之一才拆：

- 某模块需要独立扩容且占据主要资源；
- 安全/合规要求需要物理隔离；
- 团队已经形成独立所有权和发布节奏；
- 单体发布频率或故障域成为可测量瓶颈。

优先拆候选通常是地图同步、内容处理、通知或 AI 网关，不是强事务的餐厅与收藏核心。

## 19. 关键决策记录

| 决策 | v0 选择 | 原因 |
| --- | --- | --- |
| 客户端 | 微信原生小程序 | 首发只有微信，降低跨端复杂度 |
| 后端 | 模块化单体 | 兼顾交付速度和未来拆分边界 |
| POI | 腾讯位置服务适配器 | 坐标和微信生态衔接自然 |
| 核心数据 | 地图 + 运营核验 | 不依赖未授权社交内容 |
| 推荐 | 版本化确定性规则 | 可解释、可测试、可回放 |
| 社交 | v0 不做 | 降低内容审核、版权和冷启动复杂度 |
| LLM | v0 不接入 | 核心价值先由结构化数据验证 |
| 收藏 | 私有、本机可用、登录后同步 | 有复用价值但不引入社交 |
| 架构扩展 | API、适配器、领域事件、功能开关 | 支持 App、第三方和 AI 增量接入 |
| 安全稳定 | 上线门禁而非后补 | 定位和本地生活数据涉及用户信任 |
