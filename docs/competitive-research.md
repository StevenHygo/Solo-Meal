# 「一人食」微信小程序竞品与实现方式调研

> 版本：v1.0  
> 调研日期：2026-07-21  
> 范围：线下餐厅发现、单人友好筛选、餐厅评价/推荐、单人预订、社交化收藏，以及相关开放数据能力。  
> 说明：本文将公开可核验的产品能力写作“事实”；未公开的算法和技术架构只写作“合理推断”，不把推断当成厂商事实。

## 1. 结论先行

1. **需求存在，但解决方案是碎片化的。** 中国 2020 年一人户已超过 1.25 亿，占家庭户超过 25%；OpenTable 公布的美国数据也显示，2024 年单人用餐同比增长 8%。用户并非只需要“便宜餐厅”，还在意是否会被拒绝、是否有吧台/单人位、最小起点、份量、等待时间、环境压力和是否能预约 1 人。
2. **尚未发现占据主流心智的“线下单人用餐决策”垂直产品。** 国内主流产品分别擅长 POI 与交易（大众点评/美团）、位置行为（高德）、内容种草（小红书/抖音）；海外则分别擅长筛选与评论（Yelp/Google Maps）、单人库存预订（OpenTable）、明确的单人友好标签（Tabelog/Retty）和朋友图谱（Beli）。这给“一人食”留下了垂直整合机会。
3. **真正的缺口不是“附近餐厅列表”，而是可信的单人友好证据。** 地图开放接口通常能给出名称、位置、地址、电话、品类和距离，却不能稳定提供单人桌、吧台位、最低消费、单人份、独自到店占比、噪声或“一个人是否尴尬”。这些字段需要自建证据层。
4. **不建议把“爬取小红书/大众点评”作为基础数据方案。** 平台数据是否可用取决于公开接口、商务授权、内容版权和平台规则。更稳妥的首发方案是：地图 POI + 自有结构化评价 + 商家认领 + 用户主动导入链接/截图并确认；取得授权后再接入第三方内容。
5. **LLM 适合做理解和整理，不适合替代事实源。** 它可用于自然语言条件解析、截图/OCR 后的实体抽取、评价摘要、标签建议和内容审核辅助；餐厅是否有单人位、当前营业、价格和排名必须来自结构化数据并展示来源与新鲜度。
6. **推荐系统应同时展示“适合度”和“可信度”。** 一个只有 1 条好评的餐厅不应与有 30 条近期独自到店核验的餐厅同等可信。单人友好分、个性化相关性、证据量/新鲜度应分开计算和展示。

## 2. 需求与市场信号

### 2.1 国内人口结构

国家统计局提供了[第七次全国人口普查主要数据](https://www.stats.gov.cn/sj/pcsj/rkpc/d7c/)；中新网根据 2021 年《中国统计年鉴》整理指出，2020 年全国家庭户约 4.94 亿，其中“一人户”超过 1.25 亿，占比超过 25%（[来源](https://www.chinanews.com.cn/cj/2022/01-14/9652147.shtml)）。

注意：**一人户不等于每次都独自外出就餐**，独自就餐者也不都独居。该数据只能说明潜在人群基础，不能直接当作市场规模。

### 2.2 海外单人用餐行为

OpenTable 的[单人用餐专题](https://www.opentable.com/c/solo-dining/)披露：

- 2024 年美国单人用餐同比增长 8%；
- 52% 的受访美国消费者计划在当年独自用餐；
- 60% 在过去 12 个月独自在正餐餐厅吃过饭；
- OpenTable 在筛选“单人食客偏爱的餐厅”时，使用已验证用餐评价，以及评分、提前预订比例、五星评价比例、单人预订占比等行为指标。

这说明“一个人吃”既是即时决策场景，也是可以被预订和到店行为量化的场景。

### 2.3 用户问题不是单一价格问题

从产品字段和公开内容可以归纳出六类核心不确定性：

| 不确定性 | 用户真正想知道的内容 |
| --- | --- |
| 能否进店 | 是否接受 1 人、是否要求拼桌、是否高峰期拒绝单人 |
| 坐得是否舒服 | 单人桌/吧台/隔断位、座位宽度、是否正对陌生人 |
| 点餐是否合适 | 最低消费、是否有单人套餐/小份菜、是否必须点多人份 |
| 时间成本 | 排队、出餐速度、是否可预约 1 人、营业状态 |
| 心理压力 | 环境是否安静、服务是否友好、是否需要社交、是否会被催促 |
| 选择质量 | 口味、人均价、距离、卫生、真实评价、朋友或可信达人推荐 |

## 3. 竞品全景

### 3.1 对比矩阵

| 产品 | 地区/类型 | 对本项目最相关的公开能力 | 可能的实现机制 | 可借鉴点 | 主要空白 |
| --- | --- | --- | --- | --- | --- |
| 大众点评/美团 | 中国，本地生活 | 附近、品类、人均、评分、图文评价、榜单、团购/买单/排队等闭环 | POI 主数据 + 商家后台 + UGC + 交易/履约行为；具体排序未公开 | 决策字段密度、交易闭环、评价规模 | “适合一个人”的字段不够结构化；完整数据接入需授权 |
| 高德地图/扫街榜 | 中国，地图 | 周边搜索、导航；扫街榜强调导航到店、复购、专程前往等行为 | 地图 POI + 搜索/导航/到达/复访聚合行为 | 用真实行为抵抗刷评；距离和到达链路强 | 行为能说明“值得去”，不直接说明“适合一人” |
| 小红书/抖音 | 中国，内容社区 | 关键词/地点搜索、图文/短视频攻略、达人与评论互动 | 内容理解 + 社交/兴趣分发 + POI 挂接 | 能回答氛围、点什么、避坑等长尾问题 | 内容到可比较餐厅字段的转换成本高；通用内容搜索 API 不明确 |
| 饭单（App Store 检索到的新产品） | 中国，AI 收藏 | 宣称把小红书内容、点评截图等餐厅收藏到一处，由 AI 辅助选择 | 用户主动导入 + OCR/LLM 抽取 + POI 匹配（根据公开描述推断） | 绕开全网抓取，解决收藏碎片化 | 上架区域/成熟度需进一步验证；不是专门的单人到店决策 |
| Google Maps | 全球，地图 | 附近、评分、评价、照片、营业状态、繁忙时段/等待/停留时间（部分地点） | POI/商家数据 + UGC + 聚合匿名位置历史 | “现在忙不忙”是单人就餐的重要时间信号 | Google 数据在中国大陆的覆盖与接入不适合作为首发底座 |
| Yelp | 北美等，本地评价 | 距离、评分、价格、营业状态等筛选，评论、预订/配送入口 | POI + 商家页 + 大规模 UGC + 排序与反作弊 | 高密度的比较型筛选；评价质量治理 | 缺少明确的单人席位和单人份结构化字段 |
| OpenTable | 多国，预订 | 按日期、时间、地点、人数搜索；商家可按桌型和人数范围配置库存 | 餐厅桌台/班次库存 + 实时预订 + 已验证用餐评价 | “人数=1”直接查询真实可订库存；评价可与履约绑定 | 覆盖偏可预订正餐，不覆盖大量小吃/快餐；国内可用性有限 |
| Tabelog（食べログ） | 日本，餐饮评价 | `Solo dining friendly` 场景标签；座位总数、吧台席、价格区间、评论 | POI/商家字段 + 用户推荐场景标签 + UGC | 将“单人友好”设为一等筛选维度；座位类型是强证据 | 标签来源和新鲜度仍需辨别；服务重点在日本 |
| Retty | 日本，实名推荐 | 实名评价、TOP USER 推荐、地图；公开商店描述含“おひとりさまOK”；2026 年推出自然语言“咨询式”找店 | 实名社交/专家推荐 + UGC 检索 + 自然语言理解 | 可信推荐者、使用场景标签、自然语言查询结合 | 实名不自动等于真实；不专注单人场景 |
| Beli | 美国等，餐饮社交 | 餐厅排名清单与地图、朋友动态、个性化推荐、想去清单 | 用户比较/排序 + 朋友图谱 + 个性化推荐 | 用“我信任谁”替代全网平均分；收藏与评价形成循环 | 冷启动强依赖朋友数量；缺少经营/座位细节 |

### 3.2 国内产品详解

#### 大众点评/美团：信息与交易闭环最强

大众点评的公开应用描述将其定位为本地生活消费方式分享平台，包含商户/目的地信息、用户实地照片和评价、美食推荐及团购买单等能力（[App Store 页面](https://apps.apple.com/cn/app/%E5%A4%A7%E4%BC%97%E7%82%B9%E8%AF%84-%E5%90%83%E5%96%9D%E7%8E%A9%E4%B9%90%E6%8C%87%E5%8D%97/id351091731)）。

公开的[美团点评 POI 数据文档](https://poiopen.dianping.com/)列出了 POI 搜索、POI 数据、UGC、菜品、实时数据等接口文档；UGC 文档显示接口采用 `appkey`、`session`、授权码和签名鉴权（[UGC 文档](https://poiopen.dianping.com/instructions/doc/ugc.html)）。因此：

- **可确认的事实：** 存在开放接口文档和鉴权体系；
- **不可直接假设：** 新的小程序可以无条件获得完整商户评价、人均价和菜品数据；这需要按接口用途、授权范围和商务条件确认。

可借鉴：结果页信息密度、按价格/品类/距离快速收窄、评价照片和履约闭环。不可照搬：用一个总评分替代单人就餐问题。

#### 高德地图/扫街榜：从“说了什么”转向“实际去了哪里”

2025 年发布的高德扫街榜强调用户真实行为。公开报道概括的核心指标包括导航到店人数、复购率、专程前往比例和人群覆盖宽度（[新华网](http://www.news.cn/tech/20250910/12bd08e888f34380a3f8bf7199a8fd9c/c.html)、[中欧国际工商学院](https://cn.ceibs.edu/new-papers-columns/27825)）。

其启发是：推荐质量不能只靠文字好评，还可加入更难伪造的行为证据。对“一人食”而言，可以在获得用户明确同意后使用：

- 点击导航后主动确认“已独自到店”；
- 用餐后 10 秒结构化核验；
- 同一用户的再次到店；
- 收藏后到店转化。

但仅凭导航不能判断用户是否真的进店、是否独自、是否用餐，必须避免过度推断。

#### 小红书/抖音：长尾体验最丰富，数据接入最不确定

内容社区擅长表达“一个人坐哪儿”“服务员会不会奇怪”“哪道菜是单人份”等结构化平台不容易覆盖的信息。但内容通常存在：

- 一篇笔记包含多家店；
- 店名、分店、位置写法不统一；
- 软广、过时内容和跨城同名店；
- 用户收藏后难以在地图上重新查找；
- 内容和评论的转载、展示、缓存涉及授权。

检索到的[小红书官方 API 目录](https://open.xiaohongshu.com/document/api)主要展示电商开放平台的订单、售后、商品、库存和素材中心等接口，未在公开目录中确认面向任意第三方的通用笔记搜索 API。结论应是“**公开能力未确认，需商务授权**”，而不是“技术上爬得到就能使用”。

首发可行替代：让用户主动粘贴链接或上传自己保存的截图，系统只抽取餐厅候选和推荐理由，要求用户确认匹配，不批量抓取平台内容。

#### 饭单：AI 解决“收藏分散”的新方向

2026-07-21 的搜索结果中出现“饭单 - AI 餐厅收藏与美食清单”App Store 条目，描述为把小红书刷到的、点评截图的餐厅收在一处，再由 AI 辅助选择（[区域条目](https://apps.apple.com/sg/app/%E9%A5%AD%E5%8D%95-ai%E9%A4%90%E5%8E%85%E6%94%B6%E8%97%8F%E4%B8%8E%E7%BE%8E%E9%A3%9F%E6%B8%85%E5%8D%95/id6760022787?l=zh-Hans-CN)）。该条目在部分区域访问时会跳转，因此产品状态、覆盖和具体能力需进一步人工验证。

无论该产品后续状态如何，这种实现路径值得采用：**用户主动导入 -> OCR/LLM 抽取 -> POI 匹配 -> 用户确认 -> 进入自己的可搜索清单**。它比服务器批量抓取更容易建立合规和信任边界。

### 3.3 海外产品详解

#### OpenTable：以“一人可订库存”提供最强确定性

OpenTable 的商家系统允许按 party size 控制预订节奏和桌台最小/最大人数，公开术语页显示可接受人数范围从 1 开始（[Availability Planning Glossary](https://support.opentable.com/s/article/apglossary)）。这意味着用户选择 1 人后看到的是餐厅愿意释放给 1 人的真实时间库存，而不只是一个静态标签。

对“一人食”的启发：

- 有库存合作的商家，优先展示“今天 12:30 可订 1 人”；
- 没有预订系统的快餐/小吃店，用近期用户核验替代；
- “可订 1 人”是强证据，但不能等同于便宜、快速或氛围舒服。

#### Tabelog：把单人友好和吧台席做成显式字段

Tabelog 餐厅页可展示 `Solo dining friendly`、座位数和吧台席。例如公开页面会同时出现“10 席（其中 5 个吧台席）”“Counter seating”“Solo dining friendly”等字段（[示例](https://tabelog.com/en/kyoto/A2601/A260301/26006439/)）。

这是最接近本项目的字段设计：场景标签不应孤立展示，必须用座位类型、用户人数、近期核验和照片等证据解释。

#### Retty：实名推荐者 + 使用场景 + 自然语言咨询

Retty 以实名评价和 TOP USER 为产品特征（[官网](https://retty.me/)）；其 Google Play 公开描述包含“おひとりさまOK”（一人也可以）等使用场景筛选（[应用页](https://play.google.com/store/apps/details?id=me.retty&hl=ja)）。2026 年公开的新功能还支持输入菜单名和细致偏好，从大量实名评价中直接推荐匹配餐厅与相关评价（[发布信息](https://prtimes.jp/main/html/rd/p/000000238.000004025.html)）。

可借鉴：LLM/自然语言层应返回“餐厅 + 支撑它的原评价”，而不是生成一段无法溯源的推荐文案。

#### Beli：朋友关系比陌生人的平均分更有辨识度

Beli 官网公开描述包括组织餐厅清单和地图、与朋友分享、个性化推荐（[官网](https://beliapp.com/)）；应用商店描述还强调查看朋友去哪里吃、喜欢什么（[Google Play](https://play.google.com/store/apps/details?id=com.beliapp.myapp&hl=en)）。

可借鉴：

- 允许用户关注少量“口味可信的人”；
- 显示“2 位朋友独自来过并推荐”；
- 用成对比较或简单三档评价降低打分负担；
- 用“想去清单”制造下一次打开的理由。

风险：首发用户朋友很少，不能让社交关系成为获得推荐的前置条件。

#### Google Maps/Yelp：成熟筛选与行为数据

Yelp 应用公开说明可按距离、评分、价格等筛选，并拥有大量商户评价（[Google Play](https://play.google.com/store/apps/details?id=com.yelp.android&hl=en)）。Google Maps 的繁忙时段、等待时间和停留时长来自选择开启位置历史的用户所产生的聚合、匿名数据（[Google 官方说明](https://support.google.com/business/answer/6263531?hl=en)）。

两者共同说明：

- 价格、距离、营业中等是基础筛选，不是差异点；
- 匿名聚合的到店行为可辅助估计拥挤度，但需要足够样本；
- 不同平台评分口径并不一致，不应把跨平台星级简单平均。

## 4. 实现路径拆解

### 4.1 POI 与地理搜索

腾讯位置服务的[地点搜索 WebService API](https://lbs.qq.com/webservice_v1/guide-search.html)支持：

- `boundary=nearby(lat,lng,radius)` 周边搜索；
- 城市、矩形视野和多边形范围搜索；
- 按 POI 分类筛选、按距离排序；
- 返回名称、坐标、地址、电话、分类、行政区和距离等；
- 单次查询结果最多返回 200 条（以文档当前说明为准）。

这非常适合作为微信小程序首发底座，且与微信/腾讯地图生态的坐标系统衔接相对自然。但公开返回字段不包含稳定的人均价、评价、桌型或单人友好度，必须另建数据表。

### 4.2 单人友好证据层

建议把每条事实保存为独立证据，而不是直接改一个布尔标签：

| 证据 | 示例 | 可靠性建议 |
| --- | --- | --- |
| 用户近期核验 | “2026-07-18，独自到店，有 8 个吧台位” | 高；需反作弊和时间衰减 |
| 商家认领 | “接受 1 人预订，午市无最低消费” | 中高；标记为商家提供 |
| 预订履约 | 1 人预订成功并完成 | 高；合作后可用 |
| 图片证据 | 菜单中的单人套餐、吧台照片 | 中；需 OCR/人工确认 |
| 用户导入内容 | 用户上传截图并确认分店 | 中；只展示获授权部分 |
| 品类推断 | 拉面、盖饭、快餐通常适合一人 | 低；只能用于冷启动召回，不得写成事实 |

### 4.3 社交内容接入的四级方案

1. **P0，自有 UGC：** 用户在“一人食”内发布结构化体验，拥有清晰授权和可追溯来源。
2. **P0，用户主动导入：** 用户粘贴链接/上传截图，系统抽取候选餐厅，用户确认后进入私人清单；只有用户明确公开的摘要才进入公共推荐。
3. **P1，商家和创作者授权：** 商家认领页面，创作者授权同步指定内容，保留原链接、作者和时间。
4. **P2，平台商务合作：** 按平台接口协议获取 POI/评价/菜品；严格遵守展示、缓存、更新和下线要求。

不建议的路径：未授权批量爬取、绕过登录/反爬、长期缓存第三方全文和图片、把第三方评价改写后去除出处。

### 4.4 LLM 的合理角色

| 场景 | LLM 输入 | 输出 | 必要约束 |
| --- | --- | --- | --- |
| 自然语言搜索 | “静安寺附近 50 元内、安静、一个人吃的面” | 结构化筛选 JSON | 枚举/范围校验，不能直接生成店名 |
| 截图/链接导入 | OCR 文本、用户提供链接元数据 | 店名、分店、菜品、推荐理由候选 | POI 二次匹配，用户确认 |
| 评价摘要 | 检索到的自有/授权证据 | 优缺点与适合时段 | 每句话关联证据 ID，不足时明确说不知道 |
| 标签建议 | 近期结构化反馈与文本 | 吧台位、快速、安静等候选标签 | 达到票数/置信度后才公开 |
| 内容治理辅助 | 文本、图片 OCR | 广告/辱骂/隐私/虚假风险 | 机器只做初筛，高风险人工复核 |

LLM 不应承担：判断当前是否营业、捏造人均价/席位、绕过平台授权抓取内容、仅凭店名给出单人友好事实。

## 5. 机会判断与定位建议

### 5.1 建议的产品定位

> 帮你在当前位置，快速找到“一个人也舒服、吃得合适、现在去得了”的餐厅，并说明为什么可信。

“便宜”应是可选偏好，不应成为品牌的唯一含义。独自用餐也包括一人精致餐、一人旅行、一人工作餐、深夜用餐和饮食限制场景。

### 5.2 差异化组合

`地图的即时性 + Tabelog/Retty 的单人场景字段 + OpenTable 的人数确定性 + Beli 的可信关系 + AI 的内容整理`

具体表现为：

- 搜索结果默认回答“为什么适合一个人”；
- 单人友好分旁边始终显示证据量、新鲜度和来源；
- 支持用户把散落在社交平台的收藏变成自己的地点清单；
- 先服务一个城市/商圈，把证据密度做高，再扩城；
- 推荐排序使用结构化和行为证据，LLM 只负责理解与解释。

### 5.3 最小可行验证

在正式开发前，建议选一个高密度商圈做 2 周人工验证：

- 手工建档 100 家餐厅，覆盖快餐、面饭、小吃、咖啡、火锅、日料和正餐；
- 每家至少核验 6 个字段：是否接待 1 人、座位类型、单人份/最低消费、人均、午/晚高峰等待、最近核验时间；
- 邀请 30-50 位目标用户完成 100 次真实找店任务；
- 观察用户是否愿意为“单人友好证据”放弃大众点评/地图的原路径；
- 验证核心指标：搜索后 3 分钟内选定率、导航点击率、到店后符合预期率、次周复用率。

若“符合预期率”不足 70%，优先修正字段和证据质量，而不是增加更多 LLM 对话。

## 6. 风险与待验证事项

| 风险 | 影响 | 缓解方式 |
| --- | --- | --- |
| 冷启动没有单人字段 | 推荐看起来和普通地图相同 | 单商圈人工采集、商家认领、结构化用户任务 |
| 第三方数据授权失败 | 无法展示外部评价/人均 | 自有 UGC 为主；第三方只做可替换适配器 |
| 虚假评价/商家刷标签 | 失去可信度 | 履约/到店证据加权、同设备限频、异常检测、申诉 |
| LLM 幻觉或错配分店 | 推荐错误餐厅 | POI ID 为主键、候选确认、证据引用、置信度阈值 |
| 定位隐私 | 审核和用户信任风险 | 未授权可手动选位置；模糊定位优先；最小化留存 |
| 只覆盖便宜快餐 | 用户价值和商业空间过窄 | 价格是筛选项，定位是“一个人也舒服” |
| 证据过期 | 高峰政策、最低消费、席位已变化 | 证据时间衰减、90/180 天复核、用户一键报错 |
| 城市扩张稀释密度 | 有列表但没有可信答案 | 按商圈达到覆盖门槛后再扩张 |

## 7. 调研限制

- 调研基于 2026-07-21 可公开访问的网页、官方文档、应用商店描述及权威报道，未登录所有 App 逐屏测试，也未与平台商务团队确认权限和报价。
- 产品功能、开放接口、配额和平台规则会变化，正式立项前必须再次核对协议、价格、数据缓存和品牌展示条款。
- 搜索未发现成熟垂直产品不等于市场上绝对不存在小型或区域性产品；结论只表示未发现具有主流覆盖和可充分核验能力的直接竞品。
- 文中对厂商内部排序、技术架构的描述均为基于公开表现的实现推断。
