# 一人食 Web v1 更新设计

> 文档状态：提案
> 目标版本：Web v1 / 多城市 Beta
> 更新日期：2026-07-21
> 前置版本：当前 GitHub Pages Web v0

## 1. 结论先行

v1 不重做 v0。现有页面结构、安静克制的视觉风格、场景入口、筛选、列表/地图、详情、收藏和纠错流程都保留。升级重点是让当前演示版从“单商圈静态数据”变成“可持续扩展的多城市产品”。

v1 的三个核心决策：

1. **建立统一的食物类别和 SVG 图标体系。** 图标用于提高筛选和扫读效率，不改变现有信息密度，也不引入新的装饰性视觉语言。
2. **按城市中的覆盖区域逐步扩张。** 不用“已开通某城市”掩盖数据稀疏；每个商圈达到餐厅数量、证据完整度和新鲜度门槛后才对用户开放。
3. **v1 引入服务端数据库。** 多城市、空间查询、营业时间、证据版本、纠错复核和运营发布不能继续依赖 `web/data.js`。推荐 PostgreSQL + PostGIS，前端继续作为轻客户端。

LLM 不是 v1 的前置条件。它可以以后用于自然语言转筛选、授权内容抽取和证据摘要，但不能决定营业状态、价格、座位或单人友好事实。

## 2. v0 基线与不可破坏项

### 2.1 当前已经成立的体验

v0 已验证以下页面和交互：

- 当前位置或地点搜索入口；
- “现在吃、快速解决、安静坐坐、预算友好”四个场景；
- 预算、品类、营业状态、用餐时长和距离筛选；
- 餐厅列表与地图视图；
- 单人适合度、可信度、理由和核验时间；
- 餐厅详情、收藏、导航和纠错；
- 桌面与移动端响应式布局；
- 定位拒绝后的手动位置降级。

### 2.2 v1 必须保持

- 不改变首页的信息层级和主要版式；
- 不把首页改成营销落地页；
- 不增加公开社交 feed、点赞数或关注数；
- 不用一个综合星级替代单人友好证据；
- 不把 LLM 对话框设为唯一搜索入口；
- 不用更多卡片、动画或装饰稀释找店效率；
- 不在前端暴露地图服务密钥、数据库连接或管理接口；
- 未覆盖区域必须明确提示，不混入远距离或低可信结果充数。

### 2.3 v0 技术边界

当前 6 家演示餐厅全部位于 `web/data.js`，筛选和排序在 `web/app.js` 内完成，收藏与纠错只写入 `localStorage`。这一结构适用于界面验证，但存在以下扩展问题：

| 问题 | 单商圈静态版 | 多城市后的影响 |
| --- | --- | --- |
| 数据发布 | 修改 JS 后整体部署 | 无法让运营独立更新餐厅 |
| 地理搜索 | 预设距离字段 | 无法按真实坐标和范围查询 |
| 营业状态 | 手写 `openNow` | 时段、节假日和时区会错误 |
| 证据新鲜度 | 手写“7 天前” | 无法自动过期和创建复核任务 |
| 分店去重 | 人工维护少量 ID | 多供应商、连锁分店容易错配 |
| 纠错 | 本机保存 | 无法形成运营复核闭环 |
| 城市覆盖 | 页面固定文案 | 无法表达区域开通状态和数据质量 |

## 3. v1 目标与非目标

### 3.1 产品目标

- 用户可自动定位或手动选择已支持城市与商圈；
- 在已覆盖区域内，稳定返回有单人证据的真实餐厅；
- 食物类别更容易识别和筛选；
- 每条核心事实可追溯到来源、观察时间和有效期；
- 运营人员无需改代码即可导入、核验、发布、撤回餐厅；
- 城市扩张不会降低结果可信度；
- 前端在 API 或地图供应商暂时不可用时仍有清晰降级。

### 3.2 v1 非目标

- 一次性覆盖全国所有城市；
- 实时排队、实时空桌或实时客流；
- 未授权抓取大众点评、小红书、抖音等平台内容；
- 公共评论社区和关注关系；
- 复杂机器学习推荐；
- 微服务拆分；
- 全面视觉改版；
- 原生 App 或恢复微信小程序发布。

## 5. 食物类别与图标体系

### 5.1 设计原则

- 图标服务于识别，不作为插画或品牌主视觉；
- 保持 v0 的绿、珊瑚、金色点缀，不为每个品类创建一套高饱和颜色；
- 默认使用 24 x 24 SVG，`viewBox="0 0 24 24"`；
- 线宽统一为 1.75-2 px，端点和转角规则统一；
- 图标使用 `currentColor`，颜色由按钮或标签状态控制；
- 16 px、20 px、24 px 下都必须清晰；
- 图标和文字同时出现，图标不单独承担可访问名称；
- 无法识别的品类使用统一 `other` 回退图标，不临时拼接新图形。

### 5.2 首批类别字典

前端、API 和数据库只传稳定 `code`，中文名称和图标由配置表管理。

| code | 展示名称 | 图标意象 | v0 映射 |
| --- | --- | --- | --- |
| `noodles` | 面食 | 面碗 + 筷子 | 面食 |
| `rice_meal` | 米饭简餐 | 饭碗 | 粥饭中的饭类 |
| `congee` | 粥品 | 粥碗 + 勺 | 粥饭中的粥类 |
| `dumplings_buns` | 饺子包点 | 饺子轮廓 | 新增 |
| `fast_food` | 快餐小吃 | 纸袋/餐盒 | 快餐小吃 |
| `local_chinese` | 地方中餐 | 圆盘 + 筷子 | 本帮菜及其他地方菜 |
| `japanese` | 日式料理 | 定食托盘 | 日式简餐 |
| `korean` | 韩式料理 | 石锅/小菜 | 新增 |
| `hotpot` | 火锅 | 锅 + 热气 | 新增 |
| `bbq` | 烧肉烤串 | 烤网/夹子 | 烧肉 |
| `seafood` | 海鲜 | 鱼/贝 | 新增 |
| `vegetarian` | 素食轻食 | 叶片 + 餐盘 | 新增 |
| `western` | 西餐 | 刀叉 + 餐盘 | 新增 |
| `cafe_bakery` | 咖啡烘焙 | 杯 + 面包 | 新增 |
| `dessert_drinks` | 甜品饮品 | 杯/甜品勺 | 新增 |
| `other` | 其他 | 通用餐具 | 全部未知值 |

餐厅允许关联多个类别，但必须有一个 `primary_cuisine_code` 用于结果卡片和默认图标。

### 5.3 图标生成流程

1. 产品先冻结类别 code、中文名称和图形意象；
2. 优先从已采用且许可证明确的图标库中选择基础图形；
3. 面碗、粥、火锅、包点等本地化类别缺少合适图标时，再绘制自有 SVG；
4. 生成式图像工具只用于概念草图，不直接交付模糊的位图或不可维护的路径；
5. 在 Figma 或矢量工具中统一网格、线宽、留白和视觉重心；
6. 导出后使用 SVG 优化工具移除编辑器元数据、固定色值和无用分组；
7. 自动检查 `viewBox`、文件大小、`currentColor`、禁止脚本和外链；
8. 在筛选、结果卡片和移动端 16/20/24 px 三种尺寸下人工验收。

推荐目录：

```text
web/assets/cuisine/
  noodles.svg
  rice-meal.svg
  congee.svg
  dumplings-buns.svg
  fast-food.svg
  local-chinese.svg
  japanese.svg
  korean.svg
  hotpot.svg
  bbq.svg
  seafood.svg
  vegetarian.svg
  western.svg
  cafe-bakery.svg
  dessert-drinks.svg
  other.svg
```

### 5.4 图标在页面中的位置

- 品类筛选按钮：图标 + 文字；
- 搜索结果的品类文字前：16 px 图标；
- 详情页主品类：20 px 图标；
- 空结果建议：只在推荐放宽的品类旁使用；
- 地图标记继续显示单人适合度，不用食物图标替换分数；
- 首页四个场景入口保持现有符号和版式。

### 5.5 图标验收标准

- 16 个图标文件均小于 4 KB，且无嵌入图片；
- 相同尺寸下视觉重量一致；
- 不能只依靠颜色区分品类；
- 屏幕阅读器读到的是品类文字，不重复朗读装饰图标；
- 未识别 code 自动回退到 `other`；
- 图标加载失败不影响文字筛选和搜索。

## 6. 城市与覆盖区域扩展

### 6.1 不按“整座城市”一次开通

产品中的“城市已支持”应表示至少有若干连续商圈达到数据门槛，而不是只导入了城市 POI。建议采用两级模型：

```text
城市 city
  -> 覆盖区域 coverage_area（商圈、行政片区或自定义多边形）
     -> 餐厅 restaurant
```

每个覆盖区域具有状态：

| 状态 | 用户可见行为 |
| --- | --- |
| `live` | 正常搜索和推荐 |
| `beta` | 可搜索，明确显示“数据仍在补充” |
| `upcoming` | 城市选择器可见，但不返回餐厅 |
| `paused` | 暂停新增推荐，保留已收藏详情和说明 |
| `unsupported` | 提供更换位置和需求登记 |

### 6.2 建议扩张顺序

候选顺序不是最终承诺，应由用户需求、运营可达性和地图数据质量共同决定。

| 阶段 | 建议范围 | 目标 |
| --- | --- | --- |
| v1.0 | 上海：静安寺、南京西路，并扩到徐家汇/淮海中路等相邻高密度区域 | 验证数据库、区域状态和运营流程 |
| v1.1 | 北京 1-2 个办公/商业高密度区域；深圳福田或南山 1-2 个区域 | 验证跨城市时区、品类、分店与运营差异 |
| v1.2 | 杭州、广州、成都中的 1-2 城 | 只在前两城质量门槛稳定后开放 |

不建议 v1 直接显示“全国可用”。未核验城市可提供普通 POI 跳转，但不能显示单人友好分或高可信推荐。

### 6.3 区域开放门槛

区域从 `beta` 升级为 `live` 前应同时满足：

- 至少 100 家已发布餐厅；
- 用户常用 2 公里搜索范围内，80% 的测试位置能返回至少 5 家候选；
- 至少 70% 已发布餐厅在最近 90 天内核验过核心字段；
- `accepts_solo`、价格、主品类、座位类型等核心字段完整率不低于 85%；
- 高曝光餐厅分店错配率低于 1%；
- 抽样到店“符合页面描述”比例不低于 75%；
- 运营团队能在 5 个工作日内处理高优先级纠错；
- 连续两周没有严重数据质量事故。

`upcoming` 进入 `beta` 的准入规则在 Stage C 固化为 `coverage-beta-v1`：至少 30 家完成单人核验的已发布餐厅、60% 的 2 公里测试点可返回至少 5 家、80% 的 90 天核验率、85% 的核心字段完整率、90% 的 Provider ID 关联率、无高置信重复待处理，并满足抽样质量、纠错 SLA、条款/隐私评审和 PostGIS 演练。完整字段口径与运维流程见 [Stage C POI 候选与覆盖门槛运维约定](./stage-c-poi-operations.md)。门槛报告只提供证据，不自动切换区域状态。

### 6.4 城市相关 UI 更新

保持当前页面结构，只做以下增量：

- 顶部位置按钮打开“当前城市 + 最近位置 + 搜索位置”面板；
- 手动位置搜索先返回地点建议，再确认城市和坐标；
- 试点区域文案改为真实覆盖状态，例如“上海 · 静安/黄浦已覆盖”；
- 结果标题显示当前区域，但不新增大型城市欢迎页；
- 城市切换后保留预算和用餐偏好，清除不适用的距离结果；
- 超出覆盖范围时展示“此区域正在补充”，允许切换附近已覆盖区域；
- 不在未授权时自动请求精确定位。

### 6.5 地理与时间规则

- 地图供应商和中国大陆客户端统一交换 GCJ-02，同时保留供应商原始坐标和 `coord_type`；
- 数据库不得把 GCJ-02 直接标记成 EPSG:4326；空间查询使用服务端规范化后的 WGS84 `geography(Point, 4326)`，并另存 GCJ-02 数值供地图展示；
- 坐标转换集中在经过测试的服务端适配器中，API 返回给大陆地图组件前统一转换为 GCJ-02；
- 每个城市保存 IANA 时区，例如 `Asia/Shanghai`；
- 营业状态由周期营业时间、跨午夜时段、特殊日期和时区计算；
- 距离由服务端空间查询计算，不再信任静态 `distance`；
- 前端只发送用户明确选择的位置或一次性定位，不保存完整轨迹。

## 7. 是否需要数据库

### 7.1 决策

**需要。v1 多城市 Beta 开始前建立服务端数据库。**

静态 JSON 仍可作为本地演示、测试 fixture 和短期只读降级，但不能继续作为生产主数据源。

### 7.2 何时可以继续不用数据库

仅当同时满足以下条件时，静态数据仍然合理：

- 只覆盖一个商圈；
- 餐厅少于约 300 家；
- 每周或更低频率更新；
- 没有服务端纠错、运营发布和账号同步；
- 不承诺实时营业状态；
- 可以接受每次改数据都重新部署前端。

v1 的多城市、证据新鲜度和复核任务已经超出这些条件。

### 7.3 推荐技术选择

| 层 | v1 建议 | 原因 |
| --- | --- | --- |
| 数据库 | PostgreSQL 16 + PostGIS | 空间查询、事务、关系模型和成熟迁移工具 |
| API | TypeScript 模块化单体 | 与现有前端一致，避免过早微服务化 |
| 数据访问 | 显式 SQL 或轻量 ORM + 迁移工具 | 空间查询保持可控，迁移可审计 |
| 缓存 | v1 初期可不引入；需要时使用 Redis | 不让缓存成为首个复杂度来源 |
| 异步任务 | 数据库 Outbox + Worker | POI 同步、证据过期、复核任务 |
| 对象存储 | 证据图片上线后再接入 | v1 文本核验不必先引入 |
| 地图 | 高德/腾讯 Provider Adapter | 供应商可替换，密钥只在服务端 |
| 前端部署 | GitHub Pages 可继续用于 Beta | API 使用独立 HTTPS 域名和严格 CORS |

面向中国大陆正式运营时，需要评估域名备案、数据处理地域、地图条款、跨境网络和隐私要求。国际托管数据库适合原型，但不应在未评审延迟和合规前直接作为大陆生产默认选项。

## 8. v1 数据模型

### 8.1 核心表

| 表 | 作用 | 关键字段/约束 |
| --- | --- | --- |
| `cities` | 城市配置 | `code` 唯一、名称、时区、状态 |
| `coverage_areas` | 可开放区域 | `city_id`、PostGIS 多边形、状态、质量指标 |
| `cuisine_categories` | 品类字典 | `code` 唯一、名称、图标 key、排序、启用状态 |
| `restaurants` | 规范分店实体 | UUID、名称、城市、原始坐标、WGS84 空间点、地址、发布状态、版本 |
| `restaurant_cuisines` | 餐厅多品类关系 | `(restaurant_id, cuisine_id)` 唯一、是否主品类 |
| `restaurant_provider_refs` | 地图 POI 映射 | `(provider, provider_poi_id)` 唯一 |
| `restaurant_hours` | 周期与特殊营业时间 | 星期、开始/结束、特殊日期、来源 |
| `solo_profiles` | 当前单人画像快照 | 分数、可信度、评分版本、更新时间 |
| `evidence` | 字段级证据 | 属性、值、来源、观察/过期时间、状态 |
| `feedback_reports` | 用户纠错 | 类型、备注、幂等键、处理状态 |
| `curation_tasks` | 运营复核任务 | 城市、优先级、原因、负责人、截止时间 |
| `ranking_configs` | 排序权重版本 | 版本、状态、校验和、发布时间 |
| `audit_logs` | 管理审计 | 追加写、操作者、前后值、原因 |

### 8.2 关键字段变化

v0 字段应做以下规范化：

| v0 字段 | v1 处理 |
| --- | --- |
| `id: r001` | 迁移到 UUID/ULID，同时保留 `legacy_id` |
| `district` | 关联 `coverage_area_id`，展示名不承担空间判断 |
| `distance` | 不入主表，按查询位置实时计算 |
| `openNow` | 不保存为事实，按营业时间动态计算 |
| `hours` | 拆为结构化 `restaurant_hours` |
| `cuisineCode` | 迁移到品类字典和关联表 |
| `verifiedAt: 7 天前` | 保存绝对时间，前端计算相对文案 |
| `reasons` | 由后端根据证据生成 `reason_codes` |
| `soloScore` | 保存评分版本和快照，可重算和回放 |
| `evidence[]` | 迁移为独立证据行，带有效期和发布状态 |

### 8.3 索引

- `restaurants(location)` 建立 GiST 空间索引；
- `restaurants(city_id, publish_status)` 建立组合索引；
- `restaurant_cuisines(cuisine_id, restaurant_id)` 建立组合索引；
- `evidence(restaurant_id, attribute, status, expires_at)` 支持详情和过期任务；
- `curation_tasks(status, priority, due_at)` 支持运营队列；
- 文本搜索初期使用 PostgreSQL trigram/全文索引，不先引入 Elasticsearch。

## 9. API 设计

### 9.1 最小 API

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| `GET` | `/api/v1/config` | 品类、图标 key、预算、功能开关 |
| `GET` | `/api/v1/cities` | 城市与覆盖状态 |
| `GET` | `/api/v1/locations/suggest` | 地点、商圈和地铁站建议 |
| `POST` | `/api/v1/restaurants/search` | 空间搜索、筛选、排序和游标分页 |
| `GET` | `/api/v1/restaurants/:id` | 详情、证据和核验新鲜度 |
| `POST` | `/api/v1/feedback-reports` | 创建纠错和复核任务 |
| `GET` | `/api/v1/health` | 部署和依赖健康检查 |

收藏可在 v1.0 继续保存在本机。只有确定需要跨设备同步时再增加登录和收藏写 API，避免为了数据库而强迫用户注册。

### 9.2 搜索请求

```json
{
  "location": {
    "lat": 31.2231,
    "lng": 121.4452,
    "coord_type": "gcj02"
  },
  "coverage_area_id": "area_sh_jingan_01",
  "radius_m": 2000,
  "filters": {
    "budget_max_fen": 6000,
    "cuisine_codes": ["noodles"],
    "open_now": true,
    "fast_meal": false,
    "only_solo_verified": true
  },
  "sort": "recommended",
  "page_size": 20,
  "cursor": null
}
```

### 9.3 搜索响应需要新增

- `city_code` 和 `coverage_area`；
- `coverage_status`：`live/beta/upcoming/unsupported`；
- `data_freshness`；
- `ranking_version`；
- `primary_cuisine_code` 和 `cuisine_icon_key`；
- `reason_codes`，而不是前端自由拼接原因；
- `next_cursor`；
- `request_id`，便于追查错误推荐。

### 9.4 API 降级

- API 超时时保留上次成功结果并标记时间；
- 当前区域完全没有缓存时回退到位置搜索和覆盖说明；
- 地图供应商异常不影响已入库餐厅查询；
- 配置 API 失败时使用随前端发布的最后一版品类字典；
- 不把接口失败自动替换为“AI 猜测结果”。

## 10. 数据来源与运营闭环

### 10.1 允许的数据来源

| 来源 | 用途 | 发布条件 |
| --- | --- | --- |
| 地图 Provider | 名称、地址、坐标、电话、基础品类 | 保存供应商 ID 和更新时间 |
| 运营电话/现场核验 | 单人接待、座位、最低消费、用餐时间 | 记录操作者、观察时间和证据 |
| 商家公开资料/认领 | 菜单、营业时间、单人政策 | 标记“商家提供”，必要时抽查 |
| 用户结构化纠错 | 创建候选变化 | 不直接覆盖已发布事实 |
| 授权内容导入 | 菜品和体验候选 | 保留授权范围、来源链接和时间 |

不使用未授权批量抓取作为数据库基础。

### 10.2 运营流程

```text
地图召回 POI
  -> 供应商 ID/电话/名称地址去重
  -> 建立候选餐厅
  -> 运营核验核心单人字段
  -> 二次审核高曝光与疑似错配分店
  -> 发布
  -> 按曝光、证据过期和纠错生成复核任务
  -> 更新或撤回
```

### 10.3 最小运营后台

v1 后台不追求复杂仪表盘，先提供：

- 城市和覆盖区域开关；
- POI 候选导入和去重；
- 餐厅字段与证据录入；
- 草稿、待审核、已发布、已撤回状态；
- 纠错队列和任务指派；
- 证据即将过期列表；
- 修改前后值和操作者审计；
- 批量导出和失败重试。

## 11. 排序与可信度更新

v1 保留确定性、可解释排序，不上机器学习。

```text
rank_score = 0.35 * solo_fit
           + 0.25 * distance_fit
           + 0.15 * budget_fit
           + 0.15 * cuisine_fit
           + 0.10 * time_fit
```

更新点：

- 权重由 `ranking_configs` 管理，不再硬编码在浏览器；
- 硬条件先过滤，偏好条件参与排序；
- 未知字段降低可信度，不直接计为“不适合”；
- 过期证据按字段类型衰减；
- 同分按可信度、距离、稳定餐厅 ID 排序；
- 返回每个结果的 `reason_codes` 和 `ranking_version`；
- 每次配置发布可回滚，线上问题可按请求 ID 回放。

## 12. LLM 的位置

### 12.1 v1 不需要 LLM 才能上线

核心搜索应始终可以由结构化筛选、空间查询和规则排序完成。没有 LLM、LLM 超时或成本超限时，用户体验不能失效。

### 12.2 可选增强

- 把“徐家汇 60 元内安静的面”解析为地点和筛选 JSON；
- 从用户主动上传的截图/OCR 文本提取店名和候选菜品；
- 对自有或已授权证据生成短摘要；
- 为运营人员建议品类映射和疑似重复分店。

### 12.3 强制边界

- 不凭空生成餐厅；
- 不判断实时营业或排队；
- 不直接发布抽取字段；
- 每句摘要必须关联证据 ID；
- 低置信分店匹配必须由用户或运营确认；
- 不绕过第三方平台接口和内容授权。

## 13. 安全、隐私与合规

- 定位必须由用户主动触发，拒绝后仍可手动搜索；
- 精确位置只用于当前请求，日志按网格或低精度处理；
- API 密钥、数据库连接和地图服务签名只存在服务端；
- API 启用 CORS 白名单、速率限制、输入校验和稳定错误码；
- 纠错备注限制长度、字符集和频率，不渲染 HTML；
- 运营后台使用独立认证、MFA、RBAC 和审计；
- 数据库不直接暴露公网；
- 生产、预发布和开发使用不同账号、密钥和数据库；
- 正式公开服务前完成隐私政策、地图条款、域名备案和数据处理地域评审；
- 第三方内容必须记录授权依据、允许展示范围和删除机制。

## 14. 从 v0 迁移到 v1

### 14.1 迁移顺序

1. 冻结 v1 字段字典、品类 code 和图标 registry；
2. 建立数据库迁移和种子脚本；
3. 将 `web/data.js` 的 6 家餐厅导入数据库，保留 `legacy_id`；
4. 为相对时间补写绝对 `observed_at` 和 `last_verified_at`；
5. 将字符串营业时间转换为结构化时段并人工复核；
6. 实现只读配置、搜索和详情 API；
7. 前端增加 `dataSource=static|api` 功能开关；
8. 在上海试点区域同时运行静态结果和 API 结果，记录差异；
9. 差异率和性能达标后切换默认 API；
10. 保留一份经过签名或带版本号的只读快照用于短期故障降级；
11. 接入纠错和运营后台写流程；
12. 达到区域开放门槛后再增加第二城市。

### 14.2 不做大爆炸切换

- 图标体系可以先随静态数据上线；
- 数据库和 API 先只读；
- 纠错写服务在搜索稳定后开启；
- 城市按功能开关逐区开放；
- 数据迁移使用向前兼容的 expand/contract；
- 前端在迁移期兼容 v0 静态字段和 v1 DTO，但设定删除旧兼容层的日期。

## 15. 实施阶段

### 阶段 A：类别和多城市前端框架，1-2 周

- 冻结 16 个品类 code；
- 完成 SVG 图标和自动检查；
- 增加城市/覆盖区域选择状态；
- 前端改为从配置对象读取品类；
- 保持 v0 数据源和现有视觉回归测试。

退出条件：图标在桌面/移动端无布局位移，未知品类可回退，城市状态完整可演示。

### 阶段 B：数据库、只读 API 与迁移，3-4 周

- PostgreSQL/PostGIS、迁移和种子；
- 城市、覆盖、餐厅、品类、营业时间、证据表；
- 配置、地点建议、搜索和详情 API；
- 导入 6 家 v0 餐厅并做静态/API 差异测试；
- 服务端空间查询、营业计算和规则排序。

退出条件：API 结果与 v0 基准一致，P95 搜索低于 1.5 秒，错误时能回退。

### 阶段 C：上海扩区与运营后台，3-4 周

- 导入候选 POI；
- 运营去重、核验、审核和发布；
- 服务端纠错闭环；
- 证据过期任务和审计；
- 上海新增覆盖区域达到 `beta` 门槛。

退出条件：至少一个新增区域满足 beta 门槛，纠错可以在后台闭环处理。

### 阶段 D：第二城市 Beta，2-3 周

- 验证城市切换、覆盖多边形、分店去重和时区；
- 选择北京或深圳的 1-2 个区域；
- 灰度开放并监控符合预期率；
- 只有达到门槛后才升级为 `live`。

## 16. 测试与验收

### 16.1 自动化

- 品类 code、图标 registry 与数据库种子一致性；
- 所有 SVG 安全、尺寸和文件大小检查；
- PostGIS 半径与覆盖多边形查询；
- 跨午夜、节假日和时区营业状态；
- 排序版本和同分稳定性；
- API schema、错误码和游标兼容性；
- 城市切换后筛选状态；
- 定位拒绝、API 超时、空覆盖和缓存降级；
- 运营发布、撤回、纠错和审计事务；
- 桌面和移动端无横向溢出、遮挡和布局跳动。

### 16.2 v1 上线门槛

- v0 的核心页面和交互无回归；
- 所有品类都有图标、文字和回退；
- 至少 2 个城市进入 `beta`，或上海至少 3 个新增区域进入 `live`；
- 每个公开区域达到第 6.3 节的数据质量门槛；
- 搜索 API P95 <= 1.5 秒，详情 API P95 <= 800 毫秒；
- 数据库备份、恢复和迁移回滚完成演练；
- 严重分店错配率低于 1%；
- 线上纠错可以进入任务、处理并审计；
- 前端不包含生产密钥；
- 隐私、地图服务和内容授权评审通过。

### 17.1 用户价值

- 次周复用率；

- 空结果和跨区退出比例。

### 17.2 数据质量

- 每个区域已发布餐厅数；
- 核心字段完整率；

- 高优纠错处理时长；
- 证据过期餐厅占比；
- 地图 POI 与规范餐厅的去重命中率。

### 17.3 系统

- 搜索/详情 P50、P95、P99；

## 18. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 城市扩张过快 | 结果多但证据稀疏 | 按区域门槛开放，不按城市宣传目标开放 |
| 图标分类过细 | 用户找不到品类 | 首批控制在 16 类，长尾映射到上级或其他 |
| 地图条款限制缓存 | POI 成本或合规风险 | Provider Adapter、字段来源记录、正式商务确认 |
| 数据库增加运维成本 | 小团队交付变慢 | 托管 PostgreSQL、模块化单体、先不引入 Redis/微服务 |
| 营业状态错误 | 用户白跑 | 结构化时间、特殊日期、过期提示、快速纠错 |
| 分店错配 | 证据引用到错误门店 | 供应商 ID、电话、坐标和人工合并任务 |
| 运营跟不上扩张 | 数据快速过期 | 以核验产能决定开放节奏，曝光驱动复核 |
| LLM 幻觉 | 错误推荐 | LLM 只输出候选结构或带证据摘要 |
| 未授权内容 | 下架和版权风险 | 自有数据、主动导入、授权合作，不批量抓取 |
| GitHub Pages/API 跨域 | 线上请求失败 | 独立 API 域名、HTTPS、CORS 白名单、预发布验证 |

## 19. 关键决策记录

| 决策 | v1 选择 | 原因 |
| --- | --- | --- |
| 视觉 | 保留 v0 | 已验证，升级重点是数据和覆盖 |
| 图标 | 统一 SVG registry | 轻量、清晰、可主题化、可测试 |
| 城市 | 按覆盖区域开放 | 避免“全城可用”但没有可信结果 |
| 主数据 | PostgreSQL + PostGIS | 多城市空间查询和证据关系需要数据库 |
| 后端 | TypeScript 模块化单体 | 交付快，边界清晰，暂不需要微服务 |
| 缓存 | 非首发依赖 | 先证明数据库和索引足够 |
| 排序 | 服务端确定性规则 | 可解释、可版本化、可回放 |
| 收藏 | v1.0 继续本机 | 不为数据库强迫登录 |
| 社交 | 不进入 P0 | 授权、审核和冷启动成本高 |
| LLM | 可选增强 | 不让核心找店依赖模型 |
| 扩城 | 质量门槛驱动 | 可信密度比城市数量重要 |

## 20. 开发前必须确认

1. v1 首个新增上海区域和第二城市候选；
2. 每周可完成多少家餐厅的电话/现场核验；
3. 地图供应商、配额、缓存条款和生产 Key；
4. 数据库与 API 的部署区域、域名和预算；
5. 是否在 v1.0 就把纠错发送到服务端；
6. 是否需要账号与跨设备收藏；
7. 16 个品类是否覆盖目标用户的主要搜索；
8. 图标由设计师绘制、现有图标库适配还是两者结合；
9. 区域从 beta 到 live 的最终数据门槛；
10. 运营后台的负责人、审核权限和响应 SLA；
11. 隐私政策、地图展示和第三方内容授权评审负责人；
12. GitHub Pages 是否只承担 Beta，正式大陆服务是否迁移到备案域名和 CDN。

## 21. 推荐的下一步

先执行一个不依赖后端的短迭代：冻结品类字典、完成 SVG 图标、把前端品类选项改为配置驱动，并补上城市/覆盖状态的静态原型。这个迭代可以继续使用现有 v0 数据，风险低，也能为数据库 schema 和 API DTO 提供稳定输入。

随后立即开始 PostgreSQL/PostGIS 和只读 API，不建议继续向 `web/data.js` 批量增加城市餐厅。扩城数据应从第一天就进入可追溯、可核验、可撤回的服务端数据链路。
